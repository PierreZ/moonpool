# ActorHandler and Lifecycle

<!-- toc -->

Now that we understand the concept and concurrency model, let us look at the concrete trait that defines a virtual actor in moonpool.

## The ActorHandler Trait

Every actor type implements `ActorHandler`. The trait has six items, four of which have defaults:

```rust
#[async_trait(?Send)]
pub trait ActorHandler: Default + 'static {
    // Required: identifies this actor type
    fn actor_type() -> ActorType;

    // Required: handles incoming method calls
    async fn dispatch<P: Providers, C: MessageCodec>(
        &mut self,
        ctx: &ActorContext<P, C>,
        method: u32,
        body: &[u8],
    ) -> Result<Vec<u8>, ActorError>;

    // Optional: lifecycle hooks
    async fn on_activate<P, C>(&mut self, _ctx: &ActorContext<P, C>) -> Result<(), ActorError> { Ok(()) }
    async fn on_deactivate<P, C>(&mut self, _ctx: &ActorContext<P, C>) -> Result<(), ActorError> { Ok(()) }

    // Optional: placement and deactivation hints
    fn placement_strategy() -> PlacementStrategy { PlacementStrategy::Local }
    fn deactivation_hint(&self) -> DeactivationHint { DeactivationHint::KeepAlive }
}
```

The `Default` bound matters. The runtime creates actor instances with `Default::default()` before calling `on_activate`. You never construct an actor yourself.

## The Lifecycle

An actor's life follows a simple sequence:

```text
Default::default() --> on_activate() --> [dispatch()...] --> on_deactivate() --> dropped
```

**Activation** happens when the first message arrives for an identity that is not currently in memory. The runtime creates the instance, calls `on_activate`, and then dispatches the message. If `on_activate` fails, the triggering message receives an error response and the actor is not retained.

**Dispatch** handles each method call. The `method` parameter is a discriminant (1, 2, 3, ...) and `body` is the serialized request. The `#[service]` and `#[actor_impl]` macros generate this dispatch logic automatically, so in practice you write trait methods and the macro routes them.

**Deactivation** happens based on the `DeactivationHint`:

- **`KeepAlive`** (default): the actor stays in memory indefinitely, across messages.
- **`DeactivateOnIdle`**: the actor is deactivated after every message dispatch. The next message re-activates it. Useful for stateless actors or when you always want fresh state from the store.
- **`DeactivateAfterIdle(duration)`**: the actor stays cached for the given duration after the last message. If nothing arrives in that window, `on_deactivate` runs and the actor is removed. This is the Orleans default pattern, keeping hot actors in memory while reclaiming cold ones.

## A Concrete Example

Here is the banking actor from the example code. The `#[service]` macro defines the interface and generates `BankAccountRef`, while `#[actor_impl]` generates the dispatch routing:

```rust
#[service(id = 0xBA4E_4B00)]
trait BankAccount {
    async fn deposit(&mut self, req: DepositRequest) -> Result<BalanceResponse, RpcError>;
    async fn withdraw(&mut self, req: WithdrawRequest) -> Result<BalanceResponse, RpcError>;
    async fn get_balance(&mut self, req: GetBalanceRequest) -> Result<BalanceResponse, RpcError>;
}

#[derive(Default)]
struct BankAccountImpl {
    state: Option<PersistentState<BankAccountData>>,
}

#[actor_impl(BankAccount)]
impl ActorHandler for BankAccountImpl {
    fn placement_strategy() -> PlacementStrategy {
        PlacementStrategy::RoundRobin
    }

    fn deactivation_hint(&self) -> DeactivationHint {
        DeactivationHint::DeactivateAfterIdle(Duration::from_secs(30))
    }

    async fn on_activate<P: Providers, C: MessageCodec>(
        &mut self,
        ctx: &ActorContext<P, C>,
    ) -> Result<(), ActorError> {
        if let Some(store) = ctx.state_store() {
            let ps = PersistentState::<BankAccountData>::load(
                store.clone(), "BankAccount", &ctx.id.identity,
            ).await.map_err(|e| ActorError::HandlerError(format!("state load: {e}")))?;
            self.state = Some(ps);
        }
        Ok(())
    }
}
```

The `actor_type()` and `dispatch()` methods are generated by `#[actor_impl(BankAccount)]`. The `on_activate` hook loads persistent state. The deactivation hint tells the runtime to evict the actor after 30 seconds of inactivity.

## ActorContext

Every method receives an `ActorContext` that provides:

- **`ctx.id`**: the `ActorId` of the actor being called (type + identity string)
- **`ctx.router`**: an `ActorRouter` for calling other actors
- **`ctx.state_store()`**: optional access to the `ActorStateStore`
- **`ctx.actor_ref::<OtherRef>("id")`**: convenience for getting a typed reference to another actor

This means actors can call other actors during dispatch, enabling composition patterns like a transfer actor that calls deposit on one account and withdraw on another.

## The Internal Architecture

Under the hood, each registered actor type gets two levels of processing:

**Routing loop** (one per actor type): receives all `ActorMessage`s from the transport's `RequestStream` and distributes them to per-identity mailboxes. If a message arrives for an identity that does not have a mailbox yet, one is created and a new identity task is spawned.

**Identity processing loop** (one per active identity): owns the `Option<H>` actor instance. Activates on first message, dispatches messages sequentially, handles deactivation timing, and cleans up on shutdown.

When the `ActorHost` shuts down (or the owning `MoonpoolNode` calls `shutdown()`), the routing loop closes all identity mailboxes, each identity task calls `on_deactivate` on its actor, and everything drains cleanly.

## Registering Actor Types

In practice, you interact with `ActorHost` through the `MoonpoolNode` builder rather than directly:

```rust
let node = MoonpoolNode::new(cluster, config)
    .with_providers(providers)
    .register::<BankAccountImpl>()
    .start()
    .await?;
```

The `register::<BankAccountImpl>()` call records the actor type. During `start()`, the node creates the transport, router, and spawns the processing loops for each registered type.
